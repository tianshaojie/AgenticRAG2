# AGENTS.md — Agentic RAG System

## 项目概述

本项目是一个生产级 Agentic RAG（Retrieval-Augmented Generation）系统，支持文档摄入、语义检索、引用装配与 Agent 推理，所有回答必须有可追溯的 citation。

## 技术栈

| 层级 | 技术 |
|------|------|
| Backend | Python 3.12 + FastAPI + Pydantic v2 |
| Database | PostgreSQL 15+ |
| Vector Store | PostgreSQL + pgvector |
| ORM / Migration | SQLAlchemy 2.x + Alembic |
| Frontend | Vue 3 + TypeScript + Vite + shadcn-vue + Tailwind CSS |
| Testing | pytest (backend) / vitest (frontend) |
| API Contract | OpenAPI 3.1 (generated by FastAPI) |
| Config | Environment variables + `.env` files |
| Observability | Structured logging (structlog) + trace-friendly request metadata |

## 系统总原则

1. **可验证性优先于炫技** — 每个功能必须可测试、可评估、可回归
2. **模块边界优先于隐式耦合** — 所有模块通过协议接口通信
3. **保守拒答优先于无证据猜测** — 无 citation 时必须 abstain / fallback
4. **Agent 是有限状态流程** — 不允许开放式无限循环，必须有最大步数限制
5. **所有外部调用必须有 timeout / retry / structured logging**
6. **没有测试，不算完成；没有 eval，不算可靠**

## 模块职责

```
backend/app/
├── api/          # FastAPI 路由层，仅做请求验证与响应格式化
├── core/         # 配置、依赖注入、应用生命周期
├── db/           # SQLAlchemy engine、session、base model
├── schemas/      # Pydantic request/response schemas

```

## 核心接口（协议类）

所有核心组件通过 `typing.Protocol` 定义接口，实现可替换：

- `DocumentIngestor` — 解析文档、触发分块
- `Chunker` — 将文档分割为 chunks
- `Embedder` — 将文本转换为向量
- `VectorIndex` — 向量写入与检索（默认实现：pgvector）
- `Retriever` — 检索 top-k 相关 chunks
- `Reranker` — 对检索结果重排序
- `CitationAssembler` — 组装带来源的引用
- `AnswerGenerator` — 基于 context 生成回答
- `AgentPolicy` — Agent 决策策略（有限步数）
- `EvaluationRunner` — 运行 eval cases，计算指标

## 数据库设计原则

- 所有业务数据存储在 PostgreSQL
- 向量索引使用 pgvector，不引入外部向量数据库
- 向量列放在 `chunk_vectors` 表，与 `document_chunks` 关联
- 默认距离度量：**cosine similarity**（pgvector `<=>` 操作符）
- 向量维度：1536（OpenAI text-embedding-3-small 默认，可配置）
- 索引类型：**HNSW**（生产级）/ IVFFlat（开发阶段可选）

## Agent 状态机约束

```
IDLE → RETRIEVING → EVALUATING_EVIDENCE → GENERATING → DONE
                                        ↘ ABSTAINING (无足够证据)
```

- 最大循环步数：`MAX_AGENT_STEPS = 5`（可配置）
- 每步必须记录到 `agent_trace_steps` 表
- 超出步数限制时，返回 fallback 回答并标记 `abstained=true`

## 开发规范

- 所有 PR 必须包含对应测试
- API 变更必须更新 OpenAPI schema
- 数据库 schema 变更必须通过 Alembic migration
- 日志必须包含 `request_id`、`trace_id`、`user_id`（如有）
